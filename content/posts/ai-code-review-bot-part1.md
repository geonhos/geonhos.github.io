## 1. 배경 및 문제점

사내 코드 리뷰를 운영하면서 반복적으로 부딪힌 문제들입니다.

### 1-1. 일관되지 않은 리뷰 기준

- **현상**: 명확한 코딩 규칙이 존재함에도 리뷰어의 주관에 따라 피드백 방향이 달라집니다.
- **결과**: 리뷰 결과의 편차가 커져 일관된 코드 품질을 유지하기 어렵습니다.

### 1-2. 리소스 부족에 따른 품질 저하

- **현상**: 여러 프로젝트를 동시에 담당하는 리뷰어의 업무 과부하가 발생하고 있습니다.
- **결과**: 일정 압박으로 인해 리뷰가 형식적으로 진행되며 전반적인 코드 품질 검증이 저하됩니다.

### 1-3. 높은 컨텍스트 스위칭 비용

- **현상**: 적은 인원이 다양한 프로젝트를 교차 리뷰하다 보니, 생소한 코드 베이스를 파악하는 데 긴 시간이 소요됩니다.
- **결과**: 리뷰 준비 단계에서의 에너지 소모가 커 전체적인 개발 속도를 저해합니다.

### 1-4. 보안 검증의 사각지대

- **현상**: 리뷰어가 기능 로직에 집중하다 보면 하드코딩된 인증 정보나 취약한 의존성 같은 보안 이슈가 간과됩니다.
- **결과**: 보안 위협이 코드에 잔존한 채 머지되어 사후 발견 시 수정 비용이 급증합니다.

## 2. 기존 오픈소스 검토 및 자체 구축 배경

`CodiumAI PR-Agent`, `Aider` 등 AI 코드 리뷰 에이전트들이 이미 존재합니다. 하지만 사내 도입을 검토한 결과, 다음과 같은 한계점이 있었습니다.

| 한계점 | 설명 |
| :--- | :--- |
| **클라우드 API 의존성** | 대부분의 도구가 OpenAI/Anthropic 등 외부 API를 전제로 설계되어, 소스 코드 유출을 차단해야 하는 **폐쇄망 환경**에는 적용 불가 |
| **사내 컨텍스트 부재** | 우리 팀만의 코딩 컨벤션과 비즈니스 로직을 리뷰에 반영하려면 맞춤형 **RAG 연동**이 필수이나, 기존 도구는 내부 지식 통합에 취약 |
| **로컬 리소스 제약** | SWE-agent 등 로컬 구동 가능한 에이전트도 고성능 하드웨어를 요구하여, 사내 **16GB RAM** 환경에서는 운영 불가 |

기존 도구를 억지로 커스텀하느니, 차라리 직접 만드는 게 낫겠다고 판단했습니다.

## 3. 요구사항

### 보안 및 인프라

> 1-4 보안 사각지대 및 오픈소스 한계 해결

- **폐쇄망 독립 환경**: 사내망 안에서 리뷰 프로세스가 완결되어야 합니다.
- **보안 취약점 자동 점검**: 보안 위협이 있는 라이브러리 및 알려진 취약점을 사전에 필터링합니다.
- **민감 정보 노출 방지**: 소스 코드 내 하드코딩된 인증 정보(API Key 등)의 유입을 차단합니다.

### 자동화 및 품질

> 1-1 리뷰 기준 편차 및 1-2 품질 저하 해결

- **자동화된 규칙 검증**: 사전 정의된 코딩 규칙 위배 사항을 자동 감지하고, 조건 미달 시 머지(Merge)를 제한합니다.
- **테스트 및 커버리지 분석**: 유닛 테스트의 정확성을 검증하고 커버리지를 파이프라인 상에 가시화합니다.
- **성능 안티 패턴 및 코드 스멜 식별**: 비효율적인 로직을 사전에 잡아냅니다.

### 워크플로우 효율성

> 1-3 컨텍스트 스위칭 비용 해결

- **CI/CD 파이프라인 통합**: 사내 빌드/배포 파이프라인과 연동하여 리뷰어 개입을 최소화합니다.
- **사내 지식 기반 컨텍스트 제공**: 과거 PR 이력과 가이드라인을 자동으로 참조하여 리뷰어의 컨텍스트 파악 부담을 줄입니다.

## 4. 기술 스택 비교: Fine-tuning vs RAG

사내 리뷰 자동화를 위해 두 가지 접근 방식을 검토했고, **RAG(검색 증강 생성)** 방식을 채택했습니다.

| 비교 항목 | 파인튜닝 (Fine-tuning) | RAG (검색 증강 생성) |
| :--- | :--- | :--- |
| **핵심 역할** | 사내 리뷰 '스타일' 및 '응답 포맷' 체득 | 사내 최신 '지식' 및 '가이드라인' 참조 |
| **장점** | 일관된 문체와 특정 응답 포맷 출력에 유리 | 최신 문서 및 코딩 표준을 즉시 반영 가능 |
| **지식 업데이트** | 어려움 (규칙 변경 시 모델 재학습 필요) | 쉬움 (문서/PR 이력을 벡터 DB에 추가) |
| **리소스 소모** | 재학습 시 로컬 GPU 자원 및 시간 소모가 큼 | 문서 검색에 따른 토큰 처리량 증가 |
| **데이터 보안** | 학습 데이터에 민감 정보 포함 시 모델 자체 폐기 필요 | DB에서 해당 문서만 삭제하면 즉시 반영 |
| **실무 적합성** | 정적인 문법 검증 및 고정된 포맷팅에 적합 | 복잡한 비즈니스 로직 및 아키텍처 피드백에 적합 |

> **왜 RAG인가**: 컴퓨팅 리소스가 한정적이고, 프로젝트마다 기술 스택이 다릅니다. 파인튜닝은 규칙이 바뀔 때마다 재학습이 필요하지만, RAG는 DB만 업데이트하면 됩니다.
>
> LoRA/QLoRA 같은 경량 파인튜닝도 검토했지만, 가이드라인이 분기마다 바뀌는 환경에서 재학습 주기 자체가 부담이었습니다. 나중에 응답 포맷 일관성이 필요해지면 RAG + LoRA 하이브리드를 고려할 예정입니다.

## 5. LLM 모델 선정

리뷰 전용으로 확보할 수 있는 서버가 **16GB RAM, GPU 없음** 수준이라, 이 환경에서 돌릴 수 있는 7B~8B 체급 코드 특화 LLM을 비교했습니다.

### 7B~8B 체급 코딩 특화 로컬 LLM 비교

| 비교 항목 | Qwen2.5-Coder (7B) | Llama-3.1 (8B) | DeepSeek-Coder (6.7B) |
| :--- | :--- | :--- | :--- |
| **4-bit 메모리** | **약 4.7GB** | 약 4.8GB | 약 4.5GB |
| **HumanEval** | **88.4%** | 72.6% | 78.6% |
| **MBPP** | **85.5%** | 72.8% | 65.4% |
| **컨텍스트** | **128K** | 128K | 16K |

> **측정 조건**: Instruct 모델, HumanEval/MBPP pass@1 (zero-shot), FP16 기준. 4-bit 양자화 적용 시 일부 성능 하락 가능.
>
> **출처**: [Qwen2.5-Coder Technical Report](https://arxiv.org/abs/2409.12186) | [Meta Llama 3.1 Model Card](https://huggingface.co/meta-llama/Llama-3.1-8B-Instruct) | [DeepSeek-Coder GitHub](https://github.com/deepseek-ai/DeepSeek-Coder)

### 최종 선정: Qwen2.5-Coder (7B)

**Qwen2.5-Coder:7b**를 선정했습니다.

| 선정 이유 | 상세 |
| :--- | :--- |
| **벤치마크 성능** | 약 5.5조 개의 코드 토큰 학습, HumanEval 88%+ 기록 |
| **메모리 효율** | 4-bit 양자화 시 약 4.7GB로 PostgreSQL/임베딩 모델과 동시 운영 가능 |
| **컨텍스트 확장성** | 16GB 환경에서는 KV 캐시 제약으로 8~16K로 제한 운영하되, 서버 증설 시 128K까지 확장 가능 |

## 6. LLM 서빙 엔진 선정

LLM을 로컬에서 구동하려면 서빙 엔진이 필요합니다.

| 도구 | 핵심 강점 | 주 사용 목적 | 구축 난이도 |
| :--- | :--- | :--- | :--- |
| **Ollama** | Docker급 편의성 | 로컬 테스트, 사내 PoC 구축 | 매우 낮음 |
| vLLM | 대규모 트래픽 처리 | 엔터프라이즈 프로덕션, 대규모 동시 접속 | 높음 |
| Llama.cpp | 극한의 리소스 최적화 | 저사양/CPU 환경, 엣지 디바이스 | 중간 |
| LM Studio | 직관적 GUI 환경 | GUI 기반 빠른 프로토타이핑 | 매우 낮음 |

> **왜 Ollama인가**: vLLM은 GPU 필수라 탈락, Llama.cpp는 모델 관리/API를 직접 구성해야 해서 번거롭고, LM Studio는 GUI라 CI/CD 연동이 안 됩니다.
>
> Ollama는 `ollama run` 한 줄이면 모델 로딩, 양자화, REST API 서빙까지 끝납니다. 폐쇄망에서도 `ollama create`로 로컬 모델 파일을 바로 로드할 수 있습니다.

## 7. 시스템 아키텍처

### 컴포넌트 구성

![시스템 아키텍처](/images/posts/architecture.png)

### 주요 컴포넌트

| 컴포넌트 | 역할 | 선정 이유 |
| :--- | :--- | :--- |
| **Ollama** | LLM 서빙 엔진 | 모델 로딩/양자화 자동 관리, REST API 간편 호출, 폐쇄망 오프라인 구동 |
| **PostgreSQL + pgvector** | 벡터 DB | 기존 사내 PostgreSQL 인프라 재활용, 별도 벡터 DB 도입 불필요 |
| **nomic-embed-text** | 임베딩 모델 | 137M 파라미터, 약 0.3GB 메모리, 8K 컨텍스트로 긴 코드 청크 임베딩 가능 |
| **CI/CD Runner** | 파이프라인 트리거 | 기존 사내 빌드 시스템 연동, PR 이벤트 기반 자동 실행 |

### 메모리 배분 계획 (16GB)

| 항목 | 메모리 |
| :--- | :--- |
| OS + 시스템 프로세스 | 약 3GB |
| PostgreSQL + pgvector | 약 2GB |
| Ollama (Qwen2.5-Coder 7B, 4-bit) | 약 4.7GB |
| nomic-embed-text | 약 0.3GB |
| Review Service | 약 2GB |
| 여유분 (Buffer) | 약 4GB |
| **합계** | **약 16.3GB** |

### 워크플로우

1. **Trigger** -- 개발자가 코드 푸시 및 PR 생성 시 CI 파이프라인 가동
2. **Diff 추출** -- Review Service가 PR의 변경 코드(diff)를 파싱
3. **Retrieval** -- 변경 코드와 관련된 사내 코딩 컨벤션, 과거 유사 PR 리뷰 이력을 벡터 DB에서 검색
4. **Inference** -- Ollama 위의 Qwen2.5-Coder 7B가 검색된 컨텍스트와 신규 코드를 대조하여 리뷰 생성
5. **Feedback** -- 리뷰 결과를 PR 코멘트로 자동 게시

### 폐쇄망 아티팩트 관리

외부망이 없으므로 모델 파일(`.gguf`)과 Python 패키지는 보안 검수 후 USB/내부 미러로 반입합니다. Ollama 모델은 `ollama pull` 대신 `ollama create`로 로컬 파일에서 직접 로드합니다.
